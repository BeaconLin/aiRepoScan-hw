# 扫描缺陷专家评审方案

本文档描述：标注员完成单条告警标注后，该条进入「未评审」状态；经专家评审通过后方视为业务上的**最终确认**。与现有 `设计文档.md`、`接口文档.md` 中的任务、扫描结果、标注结构对齐。

---

## 1. 业务目标

| 目标 | 说明 |
|------|------|
| 职责分离 | 标注员负责初判（需要修改 / 无需修改 / 误报）；专家负责复核与确认。 |
| 状态可区分 | 「已标注」≠「已确认」：标注完成仅表示初判已提交；**评审通过**才视为最终确认。 |
| 可检索可统计 | 支持按评审状态筛选、列表展示标签，并可在任务维度统计待评审 / 已通过 / 已驳回数量。 |

---

## 2. 流程与状态机

### 2.1 角色

- **标注员**：对单条告警选择 `issueResult`（0/1/2）并可选填写 `reason`，保存后进入待评审。
- **专家**：对「未评审」条目进行通过或驳回；可选能力为「通过时是否允许修改最终结论」（见数据字段）。

### 2.2 评审状态（建议枚举）

与现有 **`annotationStatus`（是否已标注）** 分离，新增 **`reviewStatus`**：

| 值 | 含义 | 说明 |
|----|------|------|
| `0` | 未评审 | 标注保存成功后默认状态；等待专家处理。 |
| `1` | 已通过 | 专家确认，可作为最终结论（若启用 `finalIssueResult` 则以该字段为准）。 |
| `2` | 已驳回 | 退回标注员修改；再次保存标注后可重新变为未评审（规则见 4.1）。 |

可选扩展（按需）：`3` 待专家领取（若引入评审队列与认领机制）。

### 2.3 典型流转

```
[未标注] --标注员保存--> [已标注 + 未评审]
                              |
              +---------------+---------------+
              v                               v
      [已标注 + 已通过]                 [已标注 + 已驳回]
      （最终确认）                      （退回修改）
              ^                               |
              |                               |
              +--------标注员再次保存---------+
                    （重新进入未评审）
```

具体是否允许「驳回后直接由同一条标注再次提交」、是否清空专家意见等，由产品在实现时固化规则。

## 3. 数据表设计

### 3.1 现有标注主表（扩展字段）

在「一条告警对应一条标注记录」的表上增加（字段名可按后端命名规范调整为 snake_case）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `review_status` | TINYINT | 是 | 0 未评审 / 1 已通过 / 2 已驳回 |
| `reviewer_user_id` | VARCHAR | 否 | 最近一次评审人短工号 |
| `review_time` | DATETIME | 否 | 最近一次评审时间 |
| `review_comment` | VARCHAR/TEXT | 否 | 评审意见（驳回时建议业务上必填） |
| `final_issue_result` | TINYINT，可空 | 否 | **可选**：专家确认后的最终结论（0/1/2）。若不启用「专家改结论」，可通过时让该字段等于标注员的 `issue_result` 或留空仅用 `issue_result` |

**写入规则建议：**

- 标注员调用「保存标注」：写入/更新 `issue_result`、`reason` 等，**将 `review_status` 置为 `0`（未评审）**；若业务规定「修改被驳回的标注会重置评审」，可同时清空 `reviewer_user_id` / `review_time` 或将历史记入日志表（见下）。
- 专家「通过」：`review_status = 1`，填写 `reviewer_user_id`、`review_time`、`review_comment`（可选），按需写 `final_issue_result`。
- 专家「驳回」：`review_status = 2`，填写意见。

**索引建议：**

- `(task_id, review_status)`：列表筛选与待评审数量统计。
- 若已存在 `(task_id, warn_uuid)` 唯一约束，保持不变。

### 3.2 可选：审计表 `annotation_review_log`

用于合规与操作追溯（主表仅存**当前**状态）：

| 字段 | 说明 |
|------|------|
| `id` | 主键 |
| `annotation_id` 或 `task_id` + `warn_uuid` | 关联标注 |
| `action` | 如 `submit_annotation` / `approve` / `reject` |
| `operator_user_id` | 操作人短工号 |
| `comment` | 意见 |
| `created_at` | 时间 |

### 3.3 可选：任务级专家指派 `task_reviewer`

当评审权限为「按任务指派」而非全局角色时使用：

| 字段 | 说明 |
|------|------|
| `task_id` | 任务 ID |
| `reviewer_user_id` | 专家短工号 |
| `created_at` | 可选 |

唯一约束：`(task_id, reviewer_user_id)`。

若统一采用 IAM/角色「专家」且不按任务隔离，可省略此表。

## 4. 接口设计

### 4.1 修改现有接口

**保存单个缺陷标注**（现有 `POST .../api/annotations` 或等价路径）

- 成功保存后，响应体中的标注对象需包含 **`reviewStatus`**，新保存时默认为 **`0`（未评审）**。
- 若标注员在「已驳回」后修改并再次保存：服务端将 **`review_status` 重置为 `0`**（并视需求是否保留 `review_comment` 仅存在于日志表）。

**查询任务扫描结果**（`GET /api/tasks/{taskId}/scan-results`）

- 查询参数增加可选：**`reviewStatus`**（0 / 1 / 2 或约定字符串），与现有 `pageNum`、`pageSize`、`ruleName`、`annotation` 等并列。
- 响应中每条 `annotation`（或嵌套结构）增加：`reviewStatus`、`reviewerUserId`、`reviewTime`、`reviewComment`、`finalIssueResult`（若采用）。

### 4.2 新增：专家评审

**方案 A（推荐）** `POST /api/annotations/review`

请求体示例：

```json
{
  "taskId": "Tba19e29a-52c2-404f-aaba-c9e70ca3bcd3",
  "warnUuid": "6bc8163a-3d2d-4d58-a70b-f07775295c1a",
  "decision": "approve",
  "comment": "同意",
  "finalIssueResult": 1
}
```

| 字段 | 说明 |
|------|------|
| `decision` | `approve`（通过）或 `reject`（驳回） |
| `comment` | 评审意见 |
| `finalIssueResult` | 可选；仅当产品允许专家调整最终结论时传入 |

**方案 B** `PATCH /api/annotations/{annotationId}/review`  
路径使用标注主键 `id`，请求体字段同上。

**权限：** 仅专家角色或 `task_reviewer` 表中的用户可调用；否则返回 403。

### 4.3 统计接口扩展

**`GET /api/tasks/{taskId}/annotation-statistics`**（或单独 `review-statistics`）

在现有「标注完成度、标注结果分布」基础上，增加与 **评审** 相关的汇总，避免与「已标注/未标注」语义混淆，建议单独字段，例如：

- `pendingReviewCount`：未评审条数  
- `approvedReviewCount`：已通过条数  
- `rejectedReviewCount`：已驳回条数  

（或与 `annotation` 联合条件统计，由后端统一口径。）

## 5. 前端展示要点（任务详情 / 缺陷列表）

1. **列表行**  
   - 保留现有标注结论展示（需要修改 / 无需修改 / 误报）。  
   - **新增**评审状态标签：如灰色「未评审」、绿色「已通过」、橙色「已驳回」。  
   - 若有 `reviewComment`，可在展开行、Tooltip 或侧栏展示。

2. **筛选**  
   - 增加 **评审状态**：全部 / 未评审 / 已通过 / 已驳回，对应 `reviewStatus` 查询参数。

3. **统计区**  
   - 在现有图表/卡片旁增加「待评审 / 已通过 / 已驳回」数量或小型分布图，与接口 4.3 对齐。

4. **专家入口**  
   - 具备评审权限的用户：可提供 Tab 或快捷筛选 **「待我评审」**（默认 `reviewStatus=0`；若启用任务指派，再过滤本任务授权）。  
   - 行内操作：**通过** / **驳回** 弹窗，填写意见（驳回建议必填），调用 4.2 接口。

5. **文案与导出**  
   - 产品文案明确：**仅「已通过」为最终确认**。  
   - 导出或报表若只认终态，过滤条件为 `reviewStatus === 1`。

6. **前端类型与 Mock**  
   - `Annotation` / `SaveAnnotationResultData` 等 TypeScript 类型与 `src/api/task.ts` 等 Mock 数据需同步 `reviewStatus` 等字段，便于前后端联调前自测。

## 6. 与现有模型字段对照

| 现有概念 | 说明 |
|----------|------|
| `annotation.issueResult` | 标注员初判：0 需要修改，1 无需修改，2 误报 |
| `annotation.annotationStatus` | 如 1 表示已标注 |
| **新增 `reviewStatus`** | 专家评审进度：未评审 / 已通过 / 已驳回 |
